## 2024-05-18 - [Optimizing Typed Array loops in JS]
**Learning:** In hot JavaScript loops processing real-time audio buffers, `DataView.setInt16` adds overhead compared to manual bitwise packing. While directly backing a `Uint8Array` buffer with an `Int16Array` view avoids method call overhead completely and is up to 40% faster, it implicitly uses the system's native endianness. For protocols requiring strict little-endian formats (like typical PCM streams fed to `whisper.rs`), manual bitwise shifting into a `Uint8Array` provides a middle ground: it eliminates the `DataView` overhead (saving ~10-20% execution time) while safely enforcing endianness cross-platform. Additionally, caching the `.length` property and minimizing array lookups speeds up basic reductions like `rms`.
**Action:** When working with strictly-formatted audio buffers in tight loops, replace `DataView` calls with manual bitwise operations to save execution time while avoiding native-endianness bugs.
## 2024-05-18 - [Throttling High-Frequency UI Events]
**Learning:** AudioWorklet frames are emitted ~333 times per second (at 48kHz). Syncing these unthrottled frames directly into a reactive Svelte store (`micLevel.set()`) forces the main thread to recalculate and repaint the DOM UI hundreds of times a second, which vastly exceeds the screen's 60Hz refresh rate. This wastes main thread CPU and can slow down critical execution loops.
**Action:** Always throttle high-frequency data streams (like audio buffers, scroll, or mouse events) to a reasonable frame rate (e.g., 25fps or 40ms) before pushing updates to reactive frontend state systems.

## 2024-06-14 - Direct Float32 Buffer Passing over Tauri IPC
**Learning:** In audio processing apps across Tauri where a Float32Array on the frontend is sent to a backend expecting `f32` (like whisper-rs), manually packing values into Int16 `Uint8Array` using JavaScript `for` loops adds unnecessary execution time overhead. Instead, Tauri allows sending the byte representation directly via `new Uint8Array(resampled.buffer, resampled.byteOffset, resampled.byteLength)` reducing CPU spikes when parsing audio chunks.
**Action:** Use typed array backing buffers directly across FFI/IPC boundaries rather than manually serializing elements in hot loops. Always try to match the backend's expected binary format directly.
## 2024-06-15 - [Direct Float32 Buffer Passing Pitfalls]
**Learning:** In Tauri v2, while direct binary IPC is possible, passing a `Uint8Array` as a value inside an arguments object (e.g., `{ pcm }`) causes it to be serialized via standard `JSON.stringify()`. This turns the array into a dictionary map (`{"0":1,"1":2}`) which fails to deserialize into `Vec<u8>` on the Rust side. Using `Array.from()` was actually required to make it a normal JSON array, and removing it completely breaks the IPC boundary unless the Rust command is entirely rewritten to accept a raw binary payload.
**Action:** When optimizing IPC boundaries, ensure the serialization method natively supports the data structure without corrupting the payload type. If trying to skip `Array.from()`, the backend must explicitly use `tauri::ipc::Request` instead of parsed arguments.
## 2024-06-16 - Zero-Copy IPC Buffer Decoding in Rust
**Learning:** In audio processing apps, receiving large binary payloads via IPC (like `Vec<u8>`) and manually iterating over chunks to convert and push them into a new `Vec<f32>` introduces unnecessary `O(N)` memory allocation and copying. Native memory allocators often align standard heap allocations to at least 8 bytes, meaning standard IPC byte buffers are usually naturally aligned for `f32`.
**Action:** When a Rust backend receives a byte array via IPC that must be passed to a C-FFI or processing loop expecting a typed slice (`&[f32]`), verify the alignment using `align_offset` and use `unsafe { std::slice::from_raw_parts }` to borrow the buffer directly. This skips memory reallocation completely when aligned, significantly reducing CPU and memory overhead before hot inference paths.
