## 2024-05-18 - [Optimizing Typed Array loops in JS]
**Learning:** In hot JavaScript loops processing real-time audio buffers, `DataView.setInt16` adds overhead compared to manual bitwise packing. While directly backing a `Uint8Array` buffer with an `Int16Array` view avoids method call overhead completely and is up to 40% faster, it implicitly uses the system's native endianness. For protocols requiring strict little-endian formats (like typical PCM streams fed to `whisper.rs`), manual bitwise shifting into a `Uint8Array` provides a middle ground: it eliminates the `DataView` overhead (saving ~10-20% execution time) while safely enforcing endianness cross-platform. Additionally, caching the `.length` property and minimizing array lookups speeds up basic reductions like `rms`.
**Action:** When working with strictly-formatted audio buffers in tight loops, replace `DataView` calls with manual bitwise operations to save execution time while avoiding native-endianness bugs.
## 2024-05-18 - [Throttling High-Frequency UI Events]
**Learning:** AudioWorklet frames are emitted ~333 times per second (at 48kHz). Syncing these unthrottled frames directly into a reactive Svelte store (`micLevel.set()`) forces the main thread to recalculate and repaint the DOM UI hundreds of times a second, which vastly exceeds the screen's 60Hz refresh rate. This wastes main thread CPU and can slow down critical execution loops.
**Action:** Always throttle high-frequency data streams (like audio buffers, scroll, or mouse events) to a reasonable frame rate (e.g., 25fps or 40ms) before pushing updates to reactive frontend state systems.

## 2024-06-15 - [Direct Float32 Buffer Passing Pitfalls]
**Learning:** In Tauri v2, while direct binary IPC is possible, passing a `Uint8Array` as a value inside an arguments object (e.g., `{ pcm }`) causes it to be serialized via standard `JSON.stringify()`. This turns the array into a dictionary map (`{"0":1,"1":2}`) which fails to deserialize into `Vec<u8>` on the Rust side. Using `Array.from()` was actually required to make it a normal JSON array, and removing it completely breaks the IPC boundary unless the Rust command is entirely rewritten to accept a raw binary payload.
**Action:** When optimizing IPC boundaries, ensure the serialization method natively supports the data structure without corrupting the payload type. If trying to skip `Array.from()`, the backend must explicitly use `tauri::ipc::Request` instead of parsed arguments.
## 2024-06-16 - [Direct Binary IPC in Tauri v2]
**Learning:** Passing a `Uint8Array` directly to `invoke('command', pcm)` allows Tauri to skip JSON serialization, improving performance for binary payloads. However, the JS variable must be a `Uint8Array` or `ArrayBuffer`; typed arrays like `Int16Array` will still be JSON-serialized into a dictionary map. On the backend, you must use `tauri::ipc::Request<'_>` and pattern match `req.body()` against `tauri::ipc::InvokeBody::Raw(bytes)` to get the `&Vec<u8>`. The memory slice remains valid and zero-copy pointer casting like `bytes.as_ptr()` works flawlessly.
**Action:** When migrating endpoints to raw binary IPC, verify the exact JS typed array class before invoking, and use `tauri::ipc::Request` on the Rust side to safely borrow the `&Vec<u8>` without cloning.

## 2026-06-20 - [Optimizing DSP Hot Loops in JS]
**Learning:** In JavaScript hot loops processing large typed arrays (like audio linear downsampling), performing a boundary check (using a ternary operator like `hi = lo + 1 < inLen ? lo + 1 : inLen - 1`) on every single iteration adds massive overhead. Because downsampling implies the index will only exceed bounds at the very end of the array, the loop can be split to run safely up to `outLength - 1`, completely removing the branch logic. Furthermore, the standard linear interpolation math `a + (b - a) * f` is noticeably faster to execute than `a * (1 - f) + b * f`.
**Action:** When writing or reviewing DSP loops in JavaScript, always move boundary checks outside the hot loop if the maximum safe index is statically calculable, and use optimized arithmetic formulations for lerp.

## 2026-06-27 - [Direct DOM Mutation in Svelte RAF Loops]
**Learning:** In Svelte components with continuous `requestAnimationFrame` (RAF) loops (like a 60fps audio visualizer), updating reactive variables that are directly referenced in the HTML template (e.g., `style="height: {level}%"`) forces Svelte's reactivity engine to re-evaluate the component and diff the DOM on every single frame. This causes severe and continuous main thread CPU thrashing, which is especially detrimental when the app is simultaneously processing heavy operations like audio processing.
**Action:** When building high-frequency UI animations with Svelte (like audio visualizers or games), bypass the Svelte reactivity system entirely for the hot loop. Grab references to the DOM nodes using `bind:this={element}` and directly mutate their styles (e.g., `element.style.height = ...`) inside the RAF callback.
## 2024-07-28 - [Hardware Acceleration of High-Frequency UI]
**Learning:** In Svelte components (and web UIs generally), animating geometry properties like `width` or `height` at a high frequency (like an audio meter updating ~25fps or a visualizer at ~60fps) forces the browser's main thread to run expensive layout and reflow calculations on every frame, causing CPU thrashing. Also, if a property is manually updated via `requestAnimationFrame` at 60fps, having a CSS `transition` on that same property forces the browser to needlessly interpolate and immediately discard the animation between every frame.
**Action:** Always use GPU-accelerated properties (like `transform: scaleX()` or `transform: scaleY()`) instead of `width`/`height` for high-frequency animations. Furthermore, never apply CSS `transition` rules to properties that are already being updated on every frame via a continuous JavaScript loop like `requestAnimationFrame`.
## 2024-11-21 - [Hardware Acceleration in High-Frequency UIs]
**Learning:** In continuous high-frequency animation loops (like a `requestAnimationFrame` loop driving 15 audio visualizer bars at ~60fps), modifying the `height` property of DOM elements forces the browser's main thread to run expensive layout recalculations (reflows) on every frame. This wastes significant CPU cycles and can starve the main thread of resources, especially when the app is also performing other tasks (like processing audio or transcribing).
**Action:** When animating the scale/size of UI elements frequently, use the `transform: scaleX()` or `transform: scaleY()` CSS properties instead of `width` or `height`. These properties are handled by the GPU compositor layer, bypassing the main thread layout engine entirely and dropping CPU usage for the animation to near zero.
